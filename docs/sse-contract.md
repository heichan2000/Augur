# SSE Event Contract

This document is the authoritative reference for the Server-Sent Events (SSE) wire schema
used by the Augur `/chat` streaming endpoint. The backend (`backend/app/sse.py`) and the
frontend (`frontend/src/lib/sse.ts` for the types, `frontend/src/lib/sse-parser.ts` for the
wire parser) both implement this contract. Future work must not
change existing field names or types â€” only additive (backward-compatible) changes are
allowed.

---

## Wire Framing

Each event is one SSE message, UTF-8 encoded:

```
event: <type>\n
data: <compact-single-line-json>\n
\n
```

Rules:
- **`event:` line** carries the event type string (e.g. `token`).
- **`data:` line** carries a compact, single-line JSON object. No pretty-printing; no
  spaces after `:` or `,`.
- **Blank line** terminates the message (standard SSE framing).
- **No raw newlines in the `data:` line.** Newlines inside string values are JSON-escaped
  as `\n`, so a multi-line assistant token never breaks SSE framing.
- The type is carried **only** in the `event:` line; it is **not** duplicated inside the
  JSON payload.

## Event Types

### 1. `token` â€” Incremental assistant text

Emitted for each incremental chunk of text generated by the assistant.

| Field  | Type     | Description               |
|--------|----------|---------------------------|
| `text` | `string` | Partial assistant text    |

**Example** (exact bytes produced by `format_sse(TokenEvent(text="Hello"))`):

```
event: token
data: {"text":"Hello"}

```

**Escaped example** â€” text containing a newline (`\n`) and a double-quote (`"`):

Input text: `Hello\nWorld "quoted"` (two physical lines, one embedded quote)

```
event: token
data: {"text":"Hello\nWorld \"quoted\""}

```

The `data:` line is a single physical line. The embedded newline is JSON-escaped as `\n`.

---

### 2. `tool_use` â€” Assistant is invoking a tool

Emitted when the assistant decides to call a tool. The `input` object is tool-specific.

| Field   | Type               | Description                        |
|---------|--------------------|------------------------------------|
| `id`    | `string`           | Tool-call ID (provider-assigned)   |
| `name`  | `string`           | Tool name                          |
| `input` | `object`           | Tool-specific arguments            |

**Example** (exact bytes produced by `format_sse`):

```
event: tool_use
data: {"id":"call_abc","name":"search","input":{"query":"hello"}}

```

---

### 3. `error` â€” Typed error; stream ends after this

Emitted when the stream encounters an unrecoverable error. The client should treat the
stream as closed after receiving this event.

| Field     | Type     | Description                          |
|-----------|----------|--------------------------------------|
| `type`    | `string` | Stable machine-readable error code   |
| `message` | `string` | Human-readable explanation           |

**Stable `type` codes:**

| Code               | Meaning                                          |
|--------------------|--------------------------------------------------|
| `rate_limit`       | Upstream provider rate limit hit                 |
| `provider_error`   | Upstream provider returned an error              |
| `invalid_request`  | The request payload was rejected                 |
| `internal`         | Unexpected server-side failure                   |

**Example** (exact bytes):

```
event: error
data: {"type":"rate_limit","message":"Too many requests"}

```

---

### 4. `done` â€” End-of-stream sentinel

The last event of a successful turn. The client should treat the stream as complete after
receiving this event. The payload is always an empty JSON object.

| Field | Type | Description |
|-------|------|-------------|
| *(none)* | â€” | Empty payload |

**Example** (exact bytes):

```
event: done
data: {}

```

---

## Additive Evolution Rule

Future phases **MAY** add new fields to any event payload. Existing consumers **MUST**
ignore unknown fields and keep working. This is the only backward-compatible change
allowed. The following changes are **NOT** allowed:

- Renaming or removing existing fields
- Changing the type of an existing field
- Renaming event type strings
- Changing wire framing (the `event:` / `data:` / blank-line structure)
